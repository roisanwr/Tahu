## Bagian 1 — Pemahaman Proyek skorinaja

### 1.1 Problem yang Dipecahkan

Indonesia punya **64 juta UMKM**, tapi mayoritas tidak bisa akses kredit formal karena:
- Form konvensional panjang → **dropout rate tinggi**
- Bank tidak punya data keuangan formal dari segmen ini → **credit gap**

**skorinaja hadir sebagai solusi:** Mengubah proses pengajuan kredit dari *form-based* → **conversation-based** via AI chatbot, mengekstrak data terstruktur dari percakapan natural, lalu menghasilkan credit score multi-dimensional.

### 1.2 Alur Sistem (End-to-End)

```
UMKM masuk → Gemini Chatbot Interview
       ↓
   RAW Chat History (Supabase)
       ↓
   Data Extraction Pipeline (Gemini + Regex dual-extractor)
       ↓
   Validation & Enrichment (server-side rules)
       ↓
   Scoring Engine V3.1 (5 pilar)
       ↓
   Credit Score 300–850 + Rekomendasi Pinjaman
       ↓
   Dashboard UMKM (frontend Next.js)
```

## Bagian 2 — Deep-Dive: Scoring Engine V3.1

Ini core-nya. Tanpa ini tidak ada output apapun.

### 2.1 Arsitektur Pipeline Scoring

```
Input Layer → Normalisasi → Confidence Multiplier → Adjusted Scores → Aggregasi → Final Score → Post-process
```

### 2.2 5 Pilar & Bobotnya

| Pilar | Kode | Bobot | Isi |
|---|---|---|---|
| Financial (Capacity) | `S_financial` | **40%** | Profitabilitas, volume, digital cashflow |
| Collateral | `S_collateral` | **25%** | Asset coverage, lokasi tetap, foto verifikasi |
| Experience | `S_experience` | **15%** | Lama usaha, karyawan, riwayat pinjaman |
| Location | `S_location` | **10%** | Proximity pasar, densitas bisnis, infrastruktur |
| Character | `S_character` | **10%** | Marketplace reputation, sentiment, psikometrik |

### 2.3 Formula Inti

```python
# Step 1: Hitung Sub-scores (masing2 0-100)
# Step 2: Hitung Confidence Multiplier per pilar (0.50–1.00)
# Step 3: Adjusted = Sub-score × Multiplier
# Step 4: Raw_Score = Σ(Adjusted × Weight)
# Step 5: Final = clamp(round(300 + Raw_Score × 5.50), 300, 850)
# Step 6: Apply soft-cap (fraud) atau hard block (macet/contradiction)
```

### 2.4 Risk Level Mapping

| Range | Risk Level | Label |
|---|---|---|
| 750–850 | Very Low | Sangat Aman 🟢 |
| 650–749 | Low | Aman 🔵 |
| 550–649 | Medium | Sedang 🟡 |
| 450–549 | High | Berisiko 🟠 |
| 300–449 | Very High | Sangat Berisiko 🔴 |

### 2.5 Mekanisme Keamanan Scoring

- **Hard Block:** Pinjaman macet > 10jt → score langsung 380
- **Hard Block:** Contradiction count > 6 → score langsung 380
- **Soft Cap:** Fraud flag aktif → score max 579

### 2.6 Basic vs Advanced Mode

| Aspek | Basic | Advanced |
|---|---|---|
| F1 expense | Category HPP estimation | User-reported COGS+OPEX |
| F3 Digital Cashflow | Tidak aktif | Aktif (QRIS, e-wallet, bank) |
| CH3 Psychometric | Default 50 | Kuesioner psikometrik |
| Foto aset | Opsional | Wajib minimal 2 |

#### S1.3 — Normalisasi & Final Score

```python
def calculate_final_score(sub_scores, multipliers, weights) -> int:
    adjusted = [s * m for s, m in zip(sub_scores, multipliers)]
    raw_score = sum(a * w for a, w in zip(adjusted, weights))
    final = round(300 + raw_score * 5.50)
    return max(300, min(850, final))
```

## Bagian 5 — Keputusan Teknis Penting

### 5.1 Scoring Engine: Pure Functions First

Scoring engine **tidak boleh depend langsung ke database** di Sprint 1. Alasannya:
1. Bisa ditest tanpa DB ✅
2. Deterministic dan auditable ✅
3. Bisa di-reuse untuk backfill jobs ✅

### 5.2 Basic Mode Default

MVP default ke **Basic mode**. Advanced mode diaktifkan hanya kalau user secara eksplisit pilih atau upload dokumen pendukung. Ini mengurangi friction untuk UMKM yang cash-only.

### 5.3 Scope V3 (Bukan V4)

Sesuai `history.md` — keputusan sudah dibuat untuk **pakai V3.1 + scope dipersempit**:
- ✅ Bisnis yang bisa dinilai: Retail, Kuliner, Jasa bulanan stabil, Seller marketplace semi-aktif
- ⚠️ Dengan catatan: Grosir, musiman, marketplace murni
- ❌ Di luar scope: Kontraktor/project-based, petani, freelancer, enterprise > 500jt/bulan

### 5.4 Anti "Backend Slop" (dari backendrules.md)

Yang harus ada dari awal:
- ✅ Structured JSON logging + request_id
- ✅ Error taxonomy (bukan semua jadi 500)
- ✅ Pydantic config validation saat startup (fail-fast)
- ✅ Timeouts untuk semua external API calls (Gemini, Azure, Overpass)
- ✅ Rate limiting per endpoint

Yang **tidak boleh** dilakukan:
- ❌ Microservices cosplay — ini monolith, titik
- ❌ Service layer dobel tanpa alasan
- ❌ Async everywhere — pakai async hanya untuk IO-bound (DB, HTTP calls)
- ❌ Exceptions ditelan — semua error harus berbuah response yang bermakna

## Jawaban Jujur: YA, buat alur chat dulu. Ini bukan nice-to-have, ini blocker.

Ini alasannya kenapa:

### 🔗 Chat adalah sumber data scoring — bukan opsional

Scoring engine V3.1 butuh **data spesifik** untuk bisa jalan. Semua data itu **100% datang dari percakapan**. Kalau alur chat tidak didefinisikan, kita tidak skorinaja:

| Field untuk Scoring | Dapat dari... |
|---|---|
| `monthly_revenue`, `monthly_expense` | Pertanyaan keuangan |
| `transaction_frequency_daily` | Pertanyaan frekuensi transaksi |
| `assets_estimate`, `asset_type` | Pertanyaan aset/collateral |
| `years_operating`, `employee_count` | Pertanyaan profil bisnis |
| `loan_count`, `prev_loan_status` | Pertanyaan riwayat pinjaman |
| `has_wa_business`, marketplace | Pertanyaan digital presence |
| `has_fixed_location` | Pertanyaan lokasi |
| `photo_condition_score` | Upload foto usaha |
| `GPS coordinates` | Share lokasi |
| `sentiment_score`, `contradiction_count` | Diekstrak dari quality jawaban |

**Kalau salah satu pertanyaan tidak ada → field scoring kosong → multiplier drop → skor tidak akurat.**

---

### 🏗️ Alur chat nentuin 3 hal teknis yang tidak bisa diubah gampang

**1. `extracted_data` schema di `chat_history`**

DB-mu punya kolom `extracted_data JSONB` di setiap message. Schema JSON ini harus stabil. Kalau chat flow berubah tengah jalan, schema extraction ikut berubah, dan kita harus rewrite extractor.

**2. `interview_stage` di `sessions`**

DB-mu sudah define stages: `intro → profil → keuangan → dokumen → geolokasi → summary`. State machine backend-nya harus ngikutin ini persis. Kalau chat flow-nya berubah, stage order bisa berubah, dan ini breaking change ke DB constraint.

**3. Gemini System Prompt**

Ini yang paling kritikal. Quality data extraction **100% tergantung** kualitas system prompt Gemini. Kalau prompt tidak skorinaja harus tanya apa, extraction-nya juga tidak skorinaja harus ekstrak apa.

---

### 🚦 Apa yang perlu didefinisikan di "alur chat"?

Minimal ini yang perlu clear sebelum coding:

1. **Pertanyaan apa saja per stage?** (dan urutannya)
2. **Data apa yang diekstrak dari setiap jawaban?** (field name + type)
3. **Trigger apa yang mengaktifkan UI component?** (misal: setelah keuangan → muncul tombol upload foto)
4. **Kapan mode basic vs advanced bifurcate?** (pertanyaan yang berbeda)
5. **Bagaimana chatbot handle jawaban ambigu atau tidak lengkap?**
